Annotated step details
1
Define Requirements
The PRD skill conducts a structured interview covering 9 question groups to capture product overview, screens, layout, components, visual design, interactions, responsive targets, and delivery preferences.
terminal Invoke: /prd <product name>
output Output: prd/<name>-prd.md — structured markdown PRD
person Actor: You answer questions; Copilot generates the document
info Answer naturally, say "you decide" should you want defaults filled in
2
Export Task Plan
Reads the PRD and breaks it into actionable tasks across four buckets: Setup, Design, Build, and QA. Each task gets a priority and description. Output is a UTF-8 CSV ready for import.
terminal Invoke: /plan-export (auto-finds PRD) or /plan-export prd/name.md
output Output: exports/<name>-planner-tasks.csv
input Input: PRD file from Step 1
info Columns: Task Name, Bucket, Priority, Progress, Start Date, Due Date, Notes
3
Build Prototype
Generates a self-contained HTML prototype using Material Design 3 and Material Web Components via CDN. Can build from a PRD (recommended) or from a freeform description.
terminal Invoke: /material-design-prototyping build from prd/name.md
output Output: <name>.html — opens in any browser, no build tool
input Input: PRD from Step 1 or a text description
deployed_code Includes M3 color tokens, typography scale, elevation, shape, responsive breakpoints
4
Ship via Git Push
Push the prototype to a remote repository to deploy a live preview. Vercel auto-deploys on every push to main. Stakeholders can then view the prototype at the deployed URL.
terminal Commands: git add . → git commit -m "..." → git push origin main
output Output: Live URL on Vercel / Netlify / GitHub Pages
input Input: HTML prototype from Step 3
info Commit frequently for easy rollback. Each .html file gets its own URL path.
5
Collect Feedback
Injects a floating "Review" button into the HTML prototype. Stakeholders open the deployed URL, click Review, and submit approve/reject/feedback. A .md file downloads automatically.
terminal Invoke: /feedback-approval <file.html>
output Output: feedback/<name>.md with status, reviewer, date, comments
group Actors: You invoke the skill; reviewer clicks the modal in the browser
info Reviewer actions: Approve, Reject, or Provide Feedback with text comments
6
Apply Feedback & Revise
Feed the feedback .md file back into the prototyping skill. Copilot parses status and comments, applies changes to the HTML, then you push again (Step 4) and repeat until approved.
terminal Invoke: /material-design-prototyping apply changes from feedback/name.md
input Input: Feedback file from Step 5
output Output: Updated HTML prototype → loops back to Step 4 (Ship)
replay Loop: Revise → Push → Review → repeat until Approved
Support skills — can be invoked at any point in the pipeline
accessibility_new
Accessibility Audit
Audits any HTML file against WCAG 2.1 Level A+AA. Run after building a prototype (Step 3) or after applying revisions (Step 6) to catch contrast, ARIA, heading, and keyboard issues.
terminal /accessibility-audit <file.html>
summarize
Daily Report
Generates a daily project summary from git activity. Run at end-of-day to capture what was built, pushed, or revised. Useful for standups and status updates.
terminal /daily-report
build
Skill Creation
Creates new Copilot skills with SKILL.md, scripts, references, and assets. Use to extend the pipeline with additional automation or domain-specific workflows.
terminal /skill-creation
